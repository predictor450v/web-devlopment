What is MongoDB?
MongoDB is a document-oriented NoSQL database. Instead of rows in tables, it stores JSON-like documents. Each document is a self-contained unit of data — flexible, nested, and schema-optional by default.
Think of MongoDB as storing Record<string, any> objects directly to disk — except they're queryable, indexed, and clustered. 
// A MongoDB document — this IS your data model
const user = {
  _id: ObjectId("64f1a2b3c4d5e6f7a8b9c0d1"),
  name: "Ayushman",
  email: "ayushman@example.com",
  skills: ["TypeScript", "MongoDB", "React"],
  address: {             // embedded sub-document
    city: "Kolkata",
    state: "WB"
  },
  joinedAt: new Date()
};
Key Characteristics
Document-oriented Schema-flexible Horizontally scalable BSON storage Embedded documents Rich query language
Where MongoDB Fits
Use CaseWhy MongoDB?
User profilesNested data, no fixed schema
Product catalogsVarying attributes per product
Real-time analyticsFast writes, aggregation pipeline
Content managementFlexible document structure
IoT / event logsHigh write throughput
MongoDB is NOT ideal for: multi-table transactions at scale, highly relational data with complex joins, or strict schema enforcement (use PostgreSQL for those).
SQL vs NoSQL
Understanding when to pick which is more important than knowing both deeply. Here's the mental model:
🗃 SQL (Relational)
Fixed schema — tables with columns
Joins across normalized tables
ACID transactions built-in
Scales vertically (bigger server)
Great for: banking, ERP, strict data integrity
Examples: PostgreSQL, MySQL, SQLite
🍃 NoSQL (MongoDB)
Flexible schema — documents can vary
Embedded docs instead of joins
Multi-doc transactions (v4.0+)
Scales horizontally (sharding)
Great for: catalogs, CMS, real-time apps
Examples: MongoDB, DynamoDB, CouchDB

Core Concepts
BSON — What MongoDB Actually Stores
MongoDB stores data as BSON (Binary JSON) — a superset of JSON with extra types like ObjectId, Date, and Binary. You write JSON, Mongo stores BSON.
// BSON types you'll encounter in TypeScript
import { ObjectId, Binary } from "mongodb";

interface BsonExamples {
  _id: ObjectId;         // 12-byte unique ID
  createdAt: Date;       // native Date
  score: number;         // int32 / double
  active: boolean;
  avatar: Binary;        // binary blob
  tags: string[];        // arrays are native
  meta: Record<string, unknown>; // embedded doc
}
ObjectId
Every document gets a unique _id. The default ObjectId encodes timestamp + machine + process + counter — 24-char hex string.
import { ObjectId } from "mongodb";

const id = new ObjectId();
console.log(id.toString());   // "64f1a2b3c4d5e6f7a8b9c0d1"
console.log(id.getTimestamp()); // Date it was created

// Convert string → ObjectId for queries
const fromStr = new ObjectId("64f1a2b3c4d5e6f7a8b9c0d1");
Collections
A collection is like a table but with no enforced schema. Documents in the same collection can have different fields. Mongoose adds the schema layer back in TypeScript.
Rule of thumb: data that is queried together should live together. If you always fetch a user with their address, embed the address. If you query addresses independently, separate them.
Connecting to MongoDB
Option 1 — Native Driver
// db/client.ts
import { MongoClient, Db } from "mongodb";

const URI = process.env.MONGO_URI ?? "mongodb://localhost:27017";

let client: MongoClient;
let db: Db;

export async function connectDB(): Promise<Db> {
  if (db) return db;
  client = new MongoClient(URI);
  await client.connect();
  db = client.db("myapp");
  console.log("✅ MongoDB connected");
  return db;
}
Option 2 — Mongoose (Recommended for MERN)
// db/mongoose.ts
import mongoose from "mongoose";

const URI = process.env.MONGO_URI!;

export async function connectDB(): Promise<void> {
  try {
    await mongoose.connect(URI, {
      dbName: "myapp",
    });
    console.log("✅ Mongoose connected");
  } catch (err) {
    console.error("❌ DB connection failed:", err);
    process.exit(1);
  }
}

// Call in server entry point
// app.ts → connectDB().then(() => app.listen(3000))
Always store your MONGO_URI in .env and add .env to .gitignore. Never hardcode credentials.
Schema & Models with Mongoose
Mongoose brings structure to MongoDB. You define a schema → create a model → use the model to interact with a collection.
Defining a Schema (TypeScript pattern)
// models/User.ts
import { Schema, model, Document, Types } from "mongoose";

// 1. TypeScript interface (what the doc looks like)
interface IUser extends Document {
  name: string;
  email: string;
  age?: number;
  role: "admin" | "user";
  skills: string[];
  createdAt: Date;
}

// 2. Mongoose schema (validation + defaults)
const userSchema = new Schema<IUser>(
  {
    name:  { type: String,  required: true,  trim: true },
    email: { type: String,  required: true,  unique: true, lowercase: true },
    age:   { type: Number,  min: 0,  max: 120 },
    role:  { type: String,  enum: ["admin", "user"], default: "user" },
    skills: [{ type: String }],
  },
  { timestamps: true }  // auto adds createdAt, updatedAt
);

// 3. Export the model
export const User = model<IUser>("User", userSchema);

Create — Inserting Documents
save() — single document
import { User } from "./models/User";

const newUser = new User({
  name: "Ayushman",
  email: "ayushman@example.com",
  role: "user",
  skills: ["TypeScript", "React"],
});

const saved = await newUser.save();
console.log(saved._id); // ObjectId assigned by MongoDB
create() — shorthand
// Equivalent to new User({...}).save()
const user = await User.create({
  name: "Alice",
  email: "alice@example.com",
});
console.log(user.createdAt); // auto from timestamps: true
insertMany() — bulk insert
const users: IUser[] = [
  { name: "Bob",   email: "bob@x.com",   role: "user" },
  { name: "Carol", email: "carol@x.com", role: "admin" },
];

const result = await User.insertMany(users);
console.log(`Inserted ${result.length} users`);
When you call create(), Mongoose runs all validators defined in your schema before the document hits the database.
Read — Querying Documents
Basic Finders
// Find ALL documents
const all = await User.find();

// Find with filter
const admins = await User.find({ role: "admin" });

// Find ONE document
const user = await User.findOne({ email: "alice@example.com" });

// Find by _id
const byId = await User.findById("64f1a2b3c4d5e6f7a8b9c0d1");

// Select only certain fields (projection)
const names = await User.find({}).select("name email -_id");
Query Operators
// Comparison operators
await User.find({ age: { $gt: 18, $lt: 30 } });  // 18 < age < 30
await User.find({ age: { $gte: 21 } });          // age >= 21
await User.find({ role: { $in: ["admin", "mod"] } });

// Logical operators
await User.find({
  $or: [
    { role: "admin" },
    { age: { $lt: 18 } }
  ]
});

// Array query — contains element
await User.find({ skills: "TypeScript" });

// Nested field query (dot notation)
await User.find({ "address.city": "Kolkata" });

// Regex search
await User.find({ name: { $regex: /^ayu/i } });
Chaining: Sort, Limit, Skip
// Pagination pattern
const page = 2;
const limit = 10;

const users = await User
  .find({ role: "user" })
  .sort({ createdAt: -1 })   // newest first
  .skip((page - 1) * limit)
  .limit(limit)
  .lean();                    // returns plain JS objects, faster
.lean() skips Mongoose document hydration and returns plain objects. Much faster for read-only responses — use it in API GET endpoints.
Update — Modifying Documents
findByIdAndUpdate
// Returns the UPDATED document (new: true)
const updated = await User.findByIdAndUpdate(
  "64f1a2b3c4d5e6f7a8b9c0d1",
  { $set: { name: "Ayush", age: 22 } },
  { new: true, runValidators: true }
);

// Upsert: create if not exists
await User.findOneAndUpdate(
  { email: "new@x.com" },
  { $set: { name: "New User" } },
  { upsert: true, new: true }
);
Update Operators
OperatorWhat it doesExample
$setSet field value$set: { age: 25 }
$unsetRemove field$unset: { age: "" }
$incIncrement number$inc: { score: 1 }
$pushAppend to array$push: { skills: "Vue" }
$pullRemove from array$pull: { skills: "Vue" }
$addToSetPush if not exists$addToSet: { tags: "js" }
$renameRename a field$rename: { nm: "name" }
updateMany — bulk update
// Set all users with no role to "user"
const result = await User.updateMany(
  { role: { $exists: false } },
  { $set: { role: "user" } }
);
console.log(`Modified: ${result.modifiedCount}`);
Always use $set (or other operators) in updates. Passing a plain object replaces the entire document — you'll lose all unspecified fields.
Delete — Removing Documents
findByIdAndDelete
// Delete one by ID — returns deleted doc or null
const deleted = await User.findByIdAndDelete("64f1a2b...");

if (!deleted) {
  throw new Error("User not found");
}
console.log(`Deleted: ${deleted.name}`);
deleteOne / deleteMany
// Delete one matching document
await User.deleteOne({ email: "spam@x.com" });

// Delete multiple
const res = await User.deleteMany({ role: "banned" });
console.log(`Deleted: ${res.deletedCount} users`);

// Soft delete pattern (recommended)
await User.findByIdAndUpdate(id, {
  $set: { deletedAt: new Date(), isDeleted: true }
});
In production apps, prefer soft deletes (set a deletedAt field) over actually removing data. Hard deletes are irreversible and cause audit trail problems.
Aggregation Pipeline
The aggregation pipeline is MongoDB's most powerful feature. Documents flow through a series of stages, each transforming the data. Think of it as a Unix pipe for your documents.
Pipeline Stages
const result = await User.aggregate([
  // Stage 1: Filter
  { $match: { role: "user", age: { $gte: 18 } } },

  // Stage 2: Group + count
  {
    $group: {
      _id: "$address.city",
      count:    { $sum: 1 },
      avgAge:   { $avg: "$age" },
      minAge:   { $min: "$age" },
    }
  },

  // Stage 3: Sort by count desc
  { $sort: { count: -1 } },

  // Stage 4: Limit results
  { $limit: 5 },

  // Stage 5: Rename fields
  {
    $project: {
      _id: 0,
      city: "$_id",
      count: 1,
      avgAge: { $round: ["$avgAge", 1] }
    }
  }
]);
// result: [{ city: "Kolkata", count: 42, avgAge: 24.5 }, ...]
$lookup — the JOIN equivalent
// Join orders with their user
await Order.aggregate([
  {
    $lookup: {
      from: "users",         // collection to join
      localField: "userId",  // field in Order
      foreignField: "_id",  // field in User
      as: "user"             // output array field
    }
  },
  { $unwind: "$user" }       // flatten the array
]);
The aggregation pipeline is how you write complex reports, analytics queries, and data transformations in MongoDB — no separate analytics tool needed for most cases.
Indexes
Without an index, every query scans the entire collection (COLLSCAN). Indexes make reads fast at the cost of slightly slower writes and storage.
Creating Indexes in Mongoose
const userSchema = new Schema({
  email: { type: String, unique: true }, // auto creates index
  name:  { type: String, index: true },   // single field index
  age:   Number,
});

// Compound index (order matters!)
userSchema.index({ role: 1, createdAt: -1 });

// Text index for search
userSchema.index({ name: "text", bio: "text" });

// TTL index — auto-delete after 1 hour
userSchema.index(
  { createdAt: 1 },
  { expireAfterSeconds: 3600 }
);
When to Index
Index TypeUse When
Single fieldYou filter or sort by one field often
CompoundQueries use multiple fields together
TextFull-text search across string fields
UniqueField must be globally unique (email, username)
TTLDocuments should expire (sessions, OTP codes)
SparseField is optional — only index docs that have it
Don't over-index. Every index slows down writes and uses RAM. Index fields you actually query. Use explain("executionStats") to verify index usage.
Relationships
Embedding vs Referencing
Embedding (denormalized)
Store related data in same document
Single read — no join needed
Best for: one-to-few, data always read together
Example: user ↔ address
Referencing (normalized)
Store ObjectId, fetch separately
Requires populate() or $lookup
Best for: one-to-many, shared data
Example: user ↔ posts
Reference Pattern with populate()
// models/Post.ts
interface IPost extends Document {
  title: string;
  author: Types.ObjectId;  // reference to User
}
const postSchema = new Schema<IPost>({
  title:  { type: String, required: true },
  author: { type: Schema.Types.ObjectId, ref: "User" },
});
export const Post = model<IPost>("Post", postSchema);

// Fetch post WITH full author data
const post = await Post
  .findById(postId)
  .populate("author", "name email"); // only name & email
Error Handling
Mongoose Error Types
import mongoose, { Error as MongooseError } from "mongoose";

async function createUser(data: Partial<IUser>) {
  try {
    const user = await User.create(data);
    return user;
  } catch (err) {
    // Validation error (required field missing, enum violation)
    if (err instanceof MongooseError.ValidationError) {
      const messages = Object.values(err.errors)
        .map(e => e.message);
      throw new Error(`Validation: ${messages.join(", ")}`);
    }

    // Duplicate key (unique constraint)
    if ((err as any).code === 11000) {
      throw new Error("Email already registered");
    }

    // Cast error (invalid ObjectId format)
    if (err instanceof MongooseError.CastError) {
      throw new Error(`Invalid ID format: ${err.value}`);
    }

    throw err; // re-throw unknown errors
  }
}
Common Error Codes
Code / TypeMeaningFix
11000Duplicate keyCheck unique field before insert
ValidationErrorSchema validation failedCheck required fields, enums
CastErrorInvalid type (bad ObjectId)Validate input before query
MongoNetworkErrorCan't reach DBCheck URI, network, Atlas whitelist
MongoTimeoutErrorQuery took too longAdd index, paginate, or set timeout
Always handle error code 11000 explicitly in any route that creates users or unique-constraint documents — it's the most common MongoDB error in production MERN apps.
